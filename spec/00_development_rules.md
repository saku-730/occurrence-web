# 00. 開発ルール

## 目的

本ファイルは、オカレンス管理Webアプリを実装する際の共通開発ルールを定義する。  
LLMによる実装補助を前提とし、仕様逸脱・過剰実装・大きすぎる変更を避ける。

---

## 基本方針

- テスト駆動開発を絶対とする
- 仕様を遵守する
- 仕様にない機能を勝手に追加しない
- 1回の実装単位は小さくする
- 既存テストを壊してはならない
- 無関係なリファクタリングを同時に行わない
- APIを追加・変更した場合は OpenAPI 定義も更新する
- 依存クレートを追加する場合は `Cargo.toml` を直接編集する
- コメントは適宜、日本語で分かりやすく付ける

---

## TDDルール

新しい機能や仕様変更を行う場合は、以下の順序を守る。

1. 仕様を確認する
2. 最小のテストを書く
3. テストが失敗することを確認する
4. 最小実装でテストを通す
5. 必要ならリファクタリングする
6. 関連テストと全体テストを実行する
7. OpenAPIやドキュメントを更新する

---

## 実装粒度

- 1回の変更では、原則として1つの仕様だけ扱う
- 複数の関心事を混ぜない
- handler / service / repository / infrastructure を一度に大改造しない
- DB migration、DTO、service、handler、OpenAPI、テストは段階的に追加する

---

## コメント方針

コメントは、以下のような箇所に日本語で追加する。

- なぜその処理が必要なのか分かりにくい箇所
- セキュリティ上重要な処理
- RDFの主語置換やgraph検証・維持など、仕様依存の処理
- 補償処理やロールバック処理
- 監査ログの状態遷移

以下のようなコメントは不要。

- コードを読めばすぐ分かる処理の説明
- 変数名と同じ内容の説明
- 古い仕様のメモ

---

## Rustバックエンドの構成ルール

- `main.rs` は薄く保つ
- router構築は `app.rs` に寄せる
- 共有状態は `state.rs` にまとめる
- 環境設定は `config.rs` にまとめる
- 機能ごとに `features/*` に分割する
- 外部接続は `infrastructure/*` に隔離する

---

## 禁止事項

- handler に SQL を直接書かない
- handler に SPARQL を直接書かない
- handler に MinIO の詳細処理を書かない
- handler にメール送信の詳細を書かない
- service が HTTP レスポンス型に強く依存しない
- infrastructure が HTTP レスポンス型を知らない
- 認可ロジックを複数箇所に分散しない
- frontend が Jena や MinIO に直接アクセスしない
- frontend が user_id や creator を自己申告して保存できるようにしない

---

## LLM実装時の指示テンプレート

LLMに実装させるときは、以下の形で渡す。

```md
対象仕様:
- どの要件ファイルのどの項目を実装するか

制約:
- TDDで進める
- まずテストのみ書く
- 実装はまだしない
- 既存構成を崩さない
- OpenAPI更新が必要なら最後に行う

期待する出力:
- 変更対象ファイル
- 追加するテスト
- 実装方針
```
